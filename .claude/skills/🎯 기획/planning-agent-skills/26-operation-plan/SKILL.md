# Operation Plan Skill

서비스 런칭 후 운영 체계와 프로세스를 정의합니다.

## 출력 템플릿

```markdown
# {Project Name} - 운영 계획

## 1. 운영 Overview

### 운영 목표

> **"{operation_vision}"**

| 영역 | 목표 | 지표 |
|------|------|------|
| 안정성 | 99.9% Uptime | SLA |
| 성능 | < 2s 응답시간 | Latency |
| 고객 만족 | NPS > 50 | Survey |
| 효율성 | 자동화 80% | 자동화율 |

### 운영 원칙

```
1. 고객 우선: 문제 해결 최우선
2. 예방 중심: 사후 대응보다 예방
3. 데이터 기반: 감이 아닌 지표로 판단
4. 지속 개선: 매주 개선점 찾기
```

## 2. 조직 & 역할

### 운영 조직

```
┌─────────────────────────────────────────────────────────────────┐
│                        운영 조직                                  │

│                                                                  │

│      │                    │                    │                 │

│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 역할 & 책임 (RACI)

| 업무 | PM | Dev | Infra | CS |
|------|-----|-----|-------|-----|
| 인시던트 대응 | A | R | R | I |
| 기능 버그 수정 | A | R | I | I |
| 인프라 장애 | I | C | R | I |
| 고객 문의 | I | C | - | R |
| 데이터 분석 | R | C | C | I |
| 릴리즈 관리 | A | R | R | I |

*R: Responsible, A: Accountable, C: Consulted, I: Informed*

## 3. 고객 지원 (Customer Support)

### 지원 채널

| 채널 | 용도 | 응답 시간 | 담당 |
|------|------|----------|------|
| 인앱 채팅 | 즉시 문의 | < 5분 | CS |
| 이메일 | 상세 문의 | < 24시간 | CS |
| FAQ | 셀프 서비스 | 즉시 | - |
| 커뮤니티 | 사용자 간 | - | - |

### 문의 유형 & 대응

| 유형 | 비율 | 대응 | SLA |
|------|------|------|-----|
| 사용법 문의 | 40% | FAQ 안내 | < 1시간 |
| 버그 신고 | 20% | 티켓 생성 | < 4시간 |
| 기능 요청 | 15% | 백로그 추가 | < 24시간 |
| 결제 문의 | 15% | 즉시 처리 | < 1시간 |
| 불만/이탈 | 10% | 에스컬레이션 | < 30분 |

### 티켓 워크플로우

```
접수 → 분류 → 할당 → 처리 → 확인 → 종료
  │      │      │      │      │      │
자동    CS    담당자   개발/CS  고객   CS
```

### 응대 매뉴얼

| 상황 | 응대 가이드 |
|------|-----------|
| 버그 신고 | 1) 공감 표현 2) 재현 확인 3) 티켓 공유 4) 진행 상황 안내 |
| 이탈 시도 | 1) 이유 청취 2) 해결 시도 3) (필요시) 프로모션 4) 피드백 기록 |
| 기능 요청 | 1) 감사 표현 2) 배경 이해 3) 로드맵 안내 4) 업데이트 약속 |

## 4. 인시던트 관리

### 심각도 분류

| 등급 | 정의 | 예시 | 대응 시간 |
|------|------|------|----------|
| P0 | 서비스 전면 장애 | 전체 다운 | 즉시 |
| P1 | 핵심 기능 장애 | 결제 불가 | < 15분 |
| P2 | 일부 기능 장애 | 특정 기능 오류 | < 1시간 |
| P3 | 경미한 이슈 | UI 깨짐 | < 4시간 |
| P4 | 개선 사항 | 불편함 | 다음 릴리즈 |

### 인시던트 대응 프로세스

```
탐지 (Detection)
    │
    ▼
확인 (Triage)
    │
    ├─ P0/P1 → 즉시 대응팀 소집
    │
    ▼
조치 (Action)
    │
    ├─ 1차: 긴급 조치 (완화)
    ├─ 2차: 원인 파악
    └─ 3차: 근본 해결
    │
    ▼
복구 (Recovery)
    │
    ▼
포스트모템 (Postmortem)
```

### 커뮤니케이션

| 대상 | 채널 | 타이밍 | 내용 |
|------|------|--------|------|
| 내부 | Slack #incident | 즉시 | 상황, 영향, 조치 |
| 고객 | Status Page | 5분 내 | 장애 인지, 조치 중 |
| 경영진 | 이메일/전화 | P0: 즉시 | 영향, ETA |

### 포스트모템 템플릿

```markdown
## 인시던트 요약
- 발생 시각:
- 복구 시각:
- 영향 범위:
- 담당자:

## 타임라인
- HH:MM - 탐지
- HH:MM - 조치 시작
- HH:MM - 복구 완료

## 근본 원인

## 재발 방지 조치
- [ ] 조치 1
- [ ] 조치 2

## Lessons Learned
```

## 5. 배포 & 릴리즈

### 배포 전략

| 환경 | 용도 | 배포 주기 |
|------|------|----------|
| Development | 개발/테스트 | 수시 |
| Staging | QA/검증 | 매일 |
| Production | 실 서비스 | 주 1-2회 |

### 릴리즈 프로세스

```
기능 완료 → PR 리뷰 → Staging 배포 → QA → Production 배포
    │          │           │         │         │
   개발       팀원        자동화      QA     수동/자동
```

### 배포 체크리스트

- [ ] 모든 테스트 통과
- [ ] 코드 리뷰 완료
- [ ] Staging 검증 완료
- [ ] 롤백 계획 확인
- [ ] 모니터링 대시보드 준비
- [ ] 긴급 연락망 확인

### 롤백 계획

| 상황 | 기준 | 롤백 방법 |
|------|------|----------|
| 에러율 급증 | > 5% | 자동 롤백 |
| 성능 저하 | P95 > 3s | 수동 판단 |
| 비즈니스 이슈 | 결제 실패 | 즉시 롤백 |
