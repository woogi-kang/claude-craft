# Effort Estimation Skill

개발 공수를 산정하고 일정 계획을 수립합니다.

## 주의사항

```
⚠️ 공수 추정은 불확실성이 높습니다
⚠️ 버퍼(20-50%)를 반드시 포함하세요
⚠️ 실제 일정은 개발팀과 협의하세요
⚠️ 이 가이드는 참고용입니다
```

```
### 3-Point Estimation

| 시나리오 | 공수 | 확률 |
|----------|------|------|

| Most Likely | {likely}주 | 60% |

| **Expected** | **{expected}주** | - |

```

         = {expected}주

## 공수 산정 팁

### 정확도 높이기

```
✅ 작은 단위로 분해
✅ 과거 데이터 참고
✅ 팀과 함께 추정
✅ 버퍼 충분히
```

### 흔한 실수

```
❌ 버퍼 없이 추정
❌ 테스트/QA 누락
❌ 회의/커뮤니케이션 시간 누락
❌ 학습 곡선 미고려
```

## 퀄리티 체크리스트

```
□ 모든 기능이 분해되었는가?
□ FE/BE/Design/QA 모두 고려했는가?
□ 버퍼가 포함되었는가?
□ 마일스톤이 정의되었는가?
□ 리스크가 고려되었는가?
□ 비용이 추정되었는가?
□ 팀과 검토했는가?
```

## 🎯 인터랙티브 가이드

### 작성 전 확인 질문

**Q1. Feature Spec이 완성되었나요?**
- 완성됨 → 기능별 공수 분해 진행
- 미완성 → "Feature Spec 스킬을 먼저 완료해주세요"

**Q2. 추정에 참여할 팀원이 있나요?**
- 있음 → Planning Poker 등 협업 추정
- 없음 → 개인 추정 후 1.5~2배 버퍼 적용

**Q3. 과거 유사 프로젝트 데이터가 있나요?**
- 있음 → 데이터 기반 추정
- 없음 → 업계 평균 참고 + 보수적 추정

### 의사결정 포인트

| 시점 | 확인 내용 | 사용자 프롬프트 |
|------|----------|----------------|
| 기능 분해 | 단위 | "하나의 Task가 2일 이내인가요?" |
| 버퍼 적용 | 리스크 | "불확실성 높은 기능에 추가 버퍼를 넣었나요?" |
| 마일스톤 | 검증 | "각 마일스톤에서 뭘 검증할 수 있나요?" |
| 비용 | 현실성 | "이 비용으로 원하는 결과를 얻을 수 있나요?" |
