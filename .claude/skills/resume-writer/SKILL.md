# Resume Writer

회사 근무 내용을 전문적인 이력서 형태로 변환하는 Skill.

## Workflow

```
Step 1: 입력 분석 및 정보 추출
        ↓
Step 2: 누락 정보 확인 (역질문)
        ↓
Step 3: 출력 형식 확인 (경력기술서/이력서/포트폴리오)
        ↓
Step 4: 구조화된 이력서 작성
        ↓
Step 5: 검토 및 개선 제안
```

## Input Analysis

입력에서 추출할 핵심 정보:

| 카테고리 | 추출 항목 |
|---------|----------|
| 기본정보 | 회사명, 재직기간, 직급/직책 |
| 역할 | 담당 업무, 팀 내 역할, 리더십 |
| 기술 | 사용 기술, 도구, 프레임워크 |
| 성과 | 정량적 성과, 프로젝트 결과, 개선 사항 |
| 프로젝트 | 프로젝트명, 기간, 규모, 본인 기여도 |

## Required Questions

정보 부족 시 반드시 확인:

1. **재직 기간**: 시작일-종료일 (YYYY.MM 형식)
2. **직급/직책**: 정확한 명칭
3. **정량적 성과**: 숫자로 표현 가능한 성과 (%, 건수, 금액 등)
4. **본인 기여도**: 팀 프로젝트일 경우 개인 역할 명확화

정량적 성과가 없으면 추정치라도 요청. "약 30% 개선" 형태 권장.

## Output Formats

### 1. 경력기술서 (상세)

가장 상세한 형식. 각 회사/프로젝트별 깊이 있는 기술.

참조: `references/career-statement.md`

### 2. 이력서 (간결)

1-2페이지 분량의 핵심 요약. 채용 담당자용.

참조: `references/resume-format.md`

### 3. 포트폴리오용 (시각적)

프로젝트 중심의 구성. 결과물과 성과 강조.

참조: `references/portfolio-format.md`

## Writing Principles

### STAR 기법 적용

모든 성과/경험 기술 시 STAR 구조 활용:

- **S**ituation: 상황/배경
- **T**ask: 맡은 과제
- **A**ction: 수행한 행동
- **R**esult: 달성한 결과 (정량적 선호)

### 동사 선택 가이드

| 역할 유형 | 권장 동사 |
|----------|----------|
| 개발/구현 | 개발, 구현, 설계, 구축, 최적화 |
| 리딩/관리 | 리드, 주도, 관리, 조율, 멘토링 |
| 기획/분석 | 기획, 분석, 설계, 정의, 도출 |
| 개선/혁신 | 개선, 자동화, 도입, 혁신, 간소화 |
| 협업/소통 | 협업, 조율, 브릿지, 연결, 전파 |

### 숫자로 말하기

피해야 할 표현:
- "성능을 개선했습니다"
- "많은 사용자가 이용"
- "팀에 기여"

권장 표현:
- "응답 시간 40% 단축 (2초 → 1.2초)"
- "MAU 50만 서비스 운영"
- "5명 팀 리드, 3개 프로젝트 동시 진행"

## Tech Stack Formatting

기술 스택 표기 규칙:

```
[메인 기술] | [프레임워크/라이브러리] | [도구/인프라]

예시:
Flutter, Dart | Riverpod, BLoC | Firebase, GitHub Actions
Python | FastAPI, Pandas | AWS, Docker
```

## Project Description Template

```markdown
### [프로젝트명] (YYYY.MM - YYYY.MM)

**개요**: 한 줄 프로젝트 설명

**역할**: 담당 역할 (예: Flutter 개발 리드, 백엔드 개발자)

**기술**: 사용 기술 스택

**주요 성과**:
- 성과 1 (정량적 수치 포함)
- 성과 2
- 성과 3

**상세 업무**:
- 업무 내용 1
- 업무 내용 2
```

## Common Mistakes to Avoid

| 실수 | 개선 |
|-----|------|
| 업무 나열만 | → 성과와 영향력 강조 |
| 모호한 표현 | → 구체적 숫자/사례 |
| 기술만 나열 | → 기술로 해결한 문제 설명 |
| 팀 성과 = 내 성과 | → 본인 기여 구분 |
| 과거형 일관성 없음 | → 과거 경험은 과거형으로 통일 |

## Quality Checklist

작성 완료 후 확인:

- [ ] 모든 재직 기간이 명시되었는가?
- [ ] 정량적 성과가 포함되었는가?
- [ ] 기술 스택이 명확히 구분되었는가?
- [ ] 본인 역할/기여가 구체적인가?
- [ ] 동사 시제가 일관적인가?
- [ ] 불필요한 미사여구가 없는가?
- [ ] 채용 담당자가 5초 내 핵심 파악 가능한가?

## Iteration Prompt

초안 완료 후 개선 제안:

```
개선 제안
1. [더 강조할 수 있는 성과]
2. [추가하면 좋을 정량적 지표]
3. [표현 개선 포인트]

수정이 필요하신 부분이 있으시면 말씀해주세요.
```
