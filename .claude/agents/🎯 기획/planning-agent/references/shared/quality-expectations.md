# Quality Expectations

> **Version**: 1.0.0 | **Type**: Shared Reference

---

## Quality Target

> **"경험 많은 PM이 체계적으로 정리한 기획 문서" 수준**
>
> 바로 활용할 수 있는 80% 완성도. 나머지 20%는 도메인 전문성과 실제 검증.

---

## Capability Matrix

### Strong Areas (믿고 써도 됨)

| Area | Examples |
|------|----------|
| 구조화된 문서 | PRD, 기능 명세, 린 캔버스 |
| 프레임워크 기반 분석 | TAM/SAM/SOM, JTBD, MoSCoW |
| 사용자 플로우 | IA 설계, 플로우 다이어그램 |
| 기술 가이드라인 | 기술 스택 추천, 공수 추정 가이드 |
| 체크리스트/템플릿 | 법적 체크, 런칭 체크리스트 |

### Weak Areas (검토 필수)

| Area | Why | Action |
|------|-----|--------|
| 실시간 시장 데이터 | LLM 지식 컷오프 | 외부 리서치 보완 |
| 도메인 특화 규제/법률 | 전문성 한계 | 전문가 검토 필요 |
| 실제 사용자 인사이트 | 가설 기반 | 인터뷰 필요 |
| 구체적인 비용/일정 | 내부 상황 모름 | 내부 조정 필요 |

## Quality Enhancement Levels

```
Level 1: 기본 → Generic한 결과물
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
"서비스 기획해줘"

Level 2: 컨텍스트 추가 → 훨씬 나은 결과물
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Idea Intake로 문제/솔루션/타겟 정보 수집

Level 3: 피드백 루프 → 시니어 PM에 근접
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1차 결과 → "이 방향으로" → 2차 결과 → 최종
```

## Input Requirements

### Required (최소 필수)

```yaml
required:
  - 아이디어 한 줄 설명
  - 해결하려는 문제
```

### Optional (선택, 품질 향상)

```yaml
optional:
  - 타겟 사용자 힌트
  - 경쟁 서비스 이름
  - 수익화 방향
  - 기술적 제약사항
  - 예산/일정 제약
```

## Troubleshooting

| Problem | Cause | Solution |
|---------|-------|----------|
| 결과물이 generic함 | 컨텍스트 부족 | Idea Intake 충실히 |
| 시장 데이터 부정확 | 실시간 데이터 한계 | 외부 리서치 보완 |
| 기술 스택 맞지 않음 | 내부 상황 모름 | 팀 역량/제약 공유 |
| 공수 추정 안 맞음 | 복잡도 차이 | 버퍼 추가, 스프린트 단위 |
| 법적 검토 부족 | 전문성 한계 | 변호사 검토 병행 |
