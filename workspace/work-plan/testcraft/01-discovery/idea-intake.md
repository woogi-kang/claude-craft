## 1. 아이디어 한 줄 정의

**PRD/기획서를 업로드하면 플랫폼별 상세 테스트케이스를 자동 생성하는 AI 기반 SaaS**

---

## 2. Problem Statement (문제 정의)

### 2.1 핵심 문제

| 문제 | 영향 | 심각도 |
|-----|------|--------|
| QA팀의 수동 테스트케이스 작성에 과도한 시간 소요 | 개발 사이클 지연, QA 리소스 낭비 | 높음 |
| 플랫폼별 엣지케이스 누락 | 출시 후 버그 발생, 사용자 불만 | 높음 |
| 외주팀에 테스트 기준 전달의 어려움 | 품질 불일치, 커뮤니케이션 비용 증가 | 중간 |

### 2.2 문제의 근본 원인 (5 Whys 분석)

```
Q1: 왜 테스트케이스 작성에 시간이 많이 걸리나?
A1: 모든 시나리오를 수동으로 작성해야 하기 때문

Q2: 왜 수동으로 작성해야 하나?
A2: 기획서에서 테스트케이스로의 자동 변환 도구가 없기 때문

Q3: 왜 플랫폼별 엣지케이스를 놓치나?
A3: 각 플랫폼의 특수한 동작을 모두 숙지하기 어렵기 때문

Q4: 왜 외주팀 전달이 어려운가?
A4: 표준화된 테스트케이스 포맷과 기준이 없기 때문

Q5: 왜 표준화가 안 되어 있나?
A5: 테스트케이스 생성 자체가 암묵지에 의존하기 때문
```

### 2.3 문제 발생 상황

- **프로젝트 시작 시**: 기획서 기반 테스트케이스 초안 작성 (1-2주 소요)
- **기획 변경 시**: 테스트케이스 전면 수정 필요
- **멀티플랫폼 출시 시**: Android, iOS, Web, PC 각각 별도 테스트케이스 필요
- **외주 협업 시**: 테스트 기준 문서화 및 전달에 추가 시간 소요

## 3. Solution Hypothesis (솔루션 가설)

### 3.1 솔루션 개요

**AI 기반 테스트케이스 자동 생성 플랫폼**

```
[기획서/PRD 업로드]
       ↓
[AI 분석 & 파싱]
       ↓
[플랫폼 선택] → Android / iOS / Web / PC
       ↓
[테스트케이스 자동 생성]
   ├─ 기능 테스트
   ├─ 플랫폼별 엣지케이스
   ├─ 예외 시나리오
   └─ 경계값 테스트
       ↓
[Export] → Excel / Notion / Jira / TestRail
```

### 3.2 핵심 기능 가설

| 기능 | 가설 | 검증 방법 |
|-----|------|----------|
| PRD 파싱 | AI가 기획서를 이해하고 테스트 항목 추출 가능 | MVP에서 10개 기획서 테스트 |
| 플랫폼별 엣지케이스 | 사전 정의된 엣지케이스 DB로 커버 가능 | QA 전문가 검토 |
| Export 기능 | 기존 도구와의 연동이 채택 결정에 영향 | 사용자 인터뷰 |

### 3.3 플랫폼별 엣지케이스 DB (핵심 차별점)

#### Android 특화
- Back button/PopScope 처리
- 다양한 해상도 대응 (ldpi ~ xxxhdpi)
- 중국폰 호환성 (Xiaomi, Oppo, Vivo 등)
- 멀티태스킹/앱 전환 시 상태 복원
- 권한 거부 시나리오
- Battery Saver 모드 영향

#### iOS 특화
- Safe Area/Notch 대응
- 앱 전환 시 상태 유지 (Background/Foreground)
- Dynamic Island 대응 (iPhone 14 Pro+)
- Face ID/Touch ID 예외 처리
- 다크모드 전환 시 UI 검증

#### Web 특화
- 크로스 브라우저 (Chrome, Safari, Firefox, Edge)
- 반응형 브레이크포인트 (Mobile, Tablet, Desktop)
- 쿠키/세션 만료 시나리오
- 네트워크 오류/타임아웃 처리
- PWA 오프라인 모드

#### PC (Desktop) 특화
- OS별 차이 (Windows, Mac, Linux)
- 창 크기 조절 시 UI 대응
- 키보드 단축키 충돌
- 고해상도 디스플레이 (Retina) 대응
