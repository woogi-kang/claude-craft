## 1. UX 비전

### 1.1 핵심 UX 비전 선언

> **"복잡한 테스트케이스 작성을, 마치 파일 업로드처럼 쉽게"**

TestCraft의 UX는 **'전문가 도구의 결과물'을 '비전문가도 쉽게 얻을 수 있는 경험'**으로 전환하는 것을 목표로 합니다.

### 1.2 UX 목표

| 목표 | 측정 지표 | 목표치 |
|-----|----------|-------|
| **학습 용이성** | 첫 TC 생성까지 시간 | < 5분 |
| **효율성** | PRD → TC 완료 시간 | < 7분 |
| **만족도** | NPS (순추천지수) | 50+ |
| **재사용** | 주간 활성 사용률 | 70%+ |
| **오류 최소화** | TC 수정 필요 비율 | < 20% |

## 2. 타겟 사용자 UX 고려사항

### 2.1 사용자별 UX 니즈

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        사용자별 UX 니즈 매트릭스                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  QA 엔지니어 (Primary)                                                   │
│  ├─ 핵심 니즈: 시간 절약, 정확성                                         │
│  ├─ UX 기대: 전문 도구다운 상세 컨트롤                                   │
│  ├─ Pain: 복잡한 설정, 학습 곡선                                         │
│  └─ 해결: Progressive Disclosure (기본 → 고급)                          │
│                                                                          │
│  IT 기획자 (Secondary)                                                   │
│  ├─ 핵심 니즈: 간편함, 명확한 결과                                       │
│  ├─ UX 기대: 최소 입력, 최대 결과                                        │
│  ├─ Pain: 기술 용어, 복잡한 옵션                                         │
│  └─ 해결: 스마트 기본값 + 안내 문구                                      │
│                                                                          │
│  외주 개발팀 (Tertiary)                                                  │
│  ├─ 핵심 니즈: 빠른 공유, 표준화된 포맷                                  │
│  ├─ UX 기대: 바로 전달 가능한 Export                                     │
│  ├─ Pain: 클라이언트별 다른 요구                                         │
│  └─ 해결: 다양한 Export 옵션 + 커스텀 템플릿                             │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 사용자 여정별 핵심 경험

| 여정 단계 | 핵심 경험 | 감정 목표 |
|----------|---------|----------|
| **첫 방문** | 가치 즉시 이해 | "이거 나한테 필요한 거다" |
| **회원가입** | 30초 내 완료 | "간단하네" |
| **첫 TC 생성** | 5분 내 결과 확인 | "와, 진짜 되네!" |
| **결과 검토** | 품질에 놀람 | "엣지케이스까지?" |
| **Export** | 원클릭 완료 | "바로 쓸 수 있겠다" |
| **재방문** | 히스토리 유지 | "내 프로젝트 그대로네" |

## 3. UX 원칙 (Design Principles)

├─────────────────────────────────────────────────────────────────────────┤

### 3.2 원칙 적용 예시

| 원칙 | 적용 상황 | 구체적 적용 |
|-----|---------|-----------|
| Instant Value | PRD 업로드 직후 | 파일 분석 중에도 추출된 기능 실시간 표시 |
| Progressive Disclosure | 플랫폼 선택 | 기본: 플랫폼 아이콘만 / 고급: OS 버전, 디바이스 옵션 |

| Confidence Building | TC 목록 | Bulk Edit, 되돌리기, 버전 비교 제공 |
| Contextual Guidance | 첫 프로젝트 | 단계별 온보딩 투어 + 샘플 PRD 제공 |

### 5.2 피드백 인터랙션

| 상황 | 피드백 유형 | 구현 |
|-----|----------|-----|
| **버튼 클릭** | 햅틱 + 시각 | Ripple 효과, 0.1s scale-down |
| **입력 검증 성공** | 시각 | 녹색 체크 아이콘, 테두리 녹색 |
| **입력 검증 실패** | 시각 + 텍스트 | 빨간 테두리, 에러 메시지 |
| **저장 완료** | Toast | "저장되었습니다" 3초 표시 |
| **삭제 완료** | Toast + Undo | "삭제되었습니다 [되돌리기]" |
| **API 에러** | 모달 | 에러 설명 + 재시도 버튼 |
| **로딩 중** | 스피너/스켈레톤 | 버튼 내 스피너 또는 콘텐츠 스켈레톤 |

### 5.3 애니메이션 가이드라인

```yaml
timing:
  instant: 0-100ms    # 버튼 피드백, 토글
  fast: 100-200ms     # 메뉴 열기, 드롭다운
  normal: 200-300ms   # 모달, 페이지 전환
  slow: 300-500ms     # 복잡한 애니메이션
  deliberate: 500ms+  # 축하 효과, 온보딩

easing:
  default: cubic-bezier(0.4, 0, 0.2, 1)   # ease-in-out
  enter: cubic-bezier(0, 0, 0.2, 1)       # ease-out
  exit: cubic-bezier(0.4, 0, 1, 1)        # ease-in
  spring: cubic-bezier(0.175, 0.885, 0.32, 1.275)  # overshoot

principles:
  - 의미 있는 애니메이션만 사용 (장식 X)
  - 사용자 주의를 끄는 곳에만 적용
  - 접근성: prefers-reduced-motion 존중
```
