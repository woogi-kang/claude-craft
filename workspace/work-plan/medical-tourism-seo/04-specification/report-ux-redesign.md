## 1. 항목별 고객 친화적 설명 매핑 (24개 전체)

### 1.1 검색 노출 기본기

| 기술 이름 | 고객 친화적 이름 | 한 줄 설명 | 실패 시 영향 | 개선 방법 | 비유 |
|-----------|-----------------|-----------|-------------|-----------|------|
| `robots_txt` | **검색엔진 접근 허용** | 구글/네이버가 홈페이지를 읽어도 되는지 알려주는 설정 파일입니다 | 검색엔진이 홈페이지를 아예 수집하지 못해 검색 결과에 나타나지 않습니다 | 웹 개발자에게 "robots.txt 파일을 생성하고 검색엔진 접근을 허용해달라"고 요청하세요 | 건물 입구에 '영업 중' 표지판을 거는 것과 같습니다. 표지판이 없으면 손님이 문을 열지 않습니다 |
| `sitemap` | **페이지 목록 제출** | 홈페이지의 모든 페이지 목록을 검색엔진에 알려주는 파일입니다 | 검색엔진이 일부 페이지를 발견하지 못해 진료과목, 의료진 소개 등이 검색에 누락됩니다 | 웹 개발자에게 "sitemap.xml 파일을 만들고 Google Search Console에 등록해달라"고 요청하세요 | 식당 메뉴판을 입구에 붙여두는 것입니다. 메뉴판이 없으면 어떤 요리가 있는지 알 수 없습니다 |
| `meta_tags` | **검색 결과 미리보기** | 검색 결과에 보이는 제목과 설명문입니다 | 검색 결과에 의미 없는 텍스트가 표시되어 클릭률이 크게 떨어집니다 | 웹 개발자에게 "title 태그에 병원명+진료과목, description에 핵심 강점을 넣어달라"고 요청하세요 | 명함의 이름과 직함입니다. 명함에 정보가 없으면 상대방이 무엇을 하는 곳인지 모릅니다 |
| `headings` | **페이지 구조 (제목 체계)** | 페이지의 대제목-소제목 구조가 논리적인지 확인합니다 | 검색엔진이 페이지의 주제를 파악하지 못해 관련 검색어에 노출되지 않습니다 | 웹 개발자에게 "H1 태그에 페이지 핵심 주제를 1개만 넣고, H2/H3로 소주제를 정리해달라"고 요청하세요 | 신문 기사의 제목-소제목-본문 구조와 같습니다. 제목 없는 기사는 무슨 내용인지 알기 어렵습니다 |
| `canonical` | **대표 URL 지정** | 같은 내용의 페이지가 여러 주소로 접근될 때, 어떤 것이 진짜인지 알려줍니다 | 검색엔진이 같은 페이지를 중복으로 판단해 검색 순위가 분산됩니다 | 웹 개발자에게 "각 페이지에 canonical 태그를 추가해달라"고 요청하세요 | 본점 주소를 명확히 알려주는 것입니다. 같은 가게가 여러 주소로 등록되면 평점이 나뉘어 불리합니다 |
| `url_structure` | **주소(URL) 가독성** | 페이지 주소가 깔끔하고 의미 있는 단어로 구성되어 있는지 확인합니다 | 지저분한 URL은 검색엔진 크롤링 효율이 떨어지고, 공유 시 신뢰도가 낮아 보입니다 | 웹 개발자에게 "URL에 의미 있는 영문 단어를 사용하고 깊이를 3단계 이내로 유지해달라"고 요청하세요 | `hospital.kr/dermatology/acne` 처럼 주소만 봐도 내용을 알 수 있어야 합니다. `hospital.kr/page?id=3847` 은 신뢰감이 떨어집니다 |

### 1.2 사이트 속도/성능

| 기술 이름 | 고객 친화적 이름 | 한 줄 설명 | 실패 시 영향 | 개선 방법 | 비유 |
|-----------|-----------------|-----------|-------------|-----------|------|
| `lcp` | **메인 콘텐츠 로딩 속도** | 페이지에서 가장 큰 이미지나 텍스트가 화면에 나타나는 데 걸리는 시간입니다 | 2.5초 이상 걸리면 환자의 53%가 이탈합니다. 구글도 느린 사이트를 검색 순위에서 내립니다 | 이미지 최적화(WebP 변환, 압축), 서버 응답 속도 개선을 웹 개발자에게 요청하세요 | 병원 대기실에서 접수까지 걸리는 시간입니다. 너무 오래 걸리면 다른 병원으로 갑니다 |
| `inp` | **터치/클릭 반응 속도** | 버튼을 누르거나 메뉴를 클릭했을 때 반응하는 속도입니다 | 200ms 이상 걸리면 사이트가 "멈춘 것 같다"는 인상을 주어 이탈률이 높아집니다 | 불필요한 스크립트 정리, 서드파티 위젯 최적화를 웹 개발자에게 요청하세요 | 엘리베이터 버튼을 눌렀는데 불이 안 들어오면 고장인 줄 알고 계단을 찾는 것과 같습니다 |
| `cls` | **화면 밀림 현상** | 페이지가 로딩되면서 내용이 갑자기 밀려나는 정도입니다 | 환자가 "상담 예약" 버튼을 누르려는 순간 광고가 끼어들어 다른 곳을 누르게 됩니다 | 이미지/광고에 크기(width, height)를 미리 지정해 달라고 웹 개발자에게 요청하세요 | 책을 읽는데 갑자기 페이지가 바뀌는 것과 같습니다 |
| `performance_score` | **종합 성능 점수 (Lighthouse)** | 구글이 매기는 모바일 성능 종합 점수입니다 (100점 만점) | 50점 미만이면 구글 검색에서 불이익을 받고, 환자 이탈률이 급격히 증가합니다 | 이미지 최적화, 불필요한 코드 제거, 서버 속도 개선 등 종합적인 성능 튜닝이 필요합니다 | 자동차 종합검사 점수입니다. 개별 부품이 아니라 전체적인 상태를 평가합니다 |
| `mobile` | **모바일 최적화** | 스마트폰에서 홈페이지가 제대로 보이는지 확인합니다 | 모바일 미최적화 사이트는 구글 검색에서 노출이 대폭 줄어듭니다. 환자의 80%가 스마트폰으로 검색합니다 | 웹 개발자에게 "반응형 웹 디자인(viewport 설정 포함)을 적용해달라"고 요청하세요 | 전단지를 돋보기 없이 읽을 수 있어야 합니다. 글자가 너무 작으면 바로 버립니다 |

### 1.3 보안/신뢰

| 기술 이름 | 고객 친화적 이름 | 한 줄 설명 | 실패 시 영향 | 개선 방법 | 비유 |
|-----------|-----------------|-----------|-------------|-----------|------|
| `https` | **보안 연결 (자물쇠)** | 홈페이지가 암호화된 안전한 연결을 사용하는지 확인합니다 | 브라우저에 "안전하지 않음" 경고가 표시되어 환자가 개인정보 입력을 꺼립니다 | 호스팅 업체에 "SSL 인증서 설치"를 요청하세요. 무료 인증서(Let's Encrypt)도 있습니다 | 병원 출입문에 보안 시스템이 없는 것과 같습니다. 환자는 개인정보를 맡기기 불안합니다 |
| `errors_404` | **깨진 페이지 (링크 오류)** | 클릭하면 "페이지를 찾을 수 없습니다" 오류가 나는 링크입니다 | 환자가 진료 안내, 예약 페이지에 접근하지 못해 바로 이탈합니다. 검색엔진 평가도 하락합니다 | 웹 개발자에게 깨진 링크 목록을 전달하고 수정 또는 삭제를 요청하세요 | 안내 데스크에서 "3층 진료실"이라 했는데 3층이 공사 중인 것과 같습니다 |
| `images_alt` | **이미지 설명 텍스트** | 이미지에 대체 텍스트(설명)가 있는지 확인합니다 | 검색엔진이 이미지 내용을 파악하지 못하고, 시각장애인 환자가 정보를 얻을 수 없습니다 | 웹 개발자에게 "모든 이미지에 내용을 설명하는 alt 텍스트를 넣어달라"고 요청하세요 | 사진 뒷면에 메모를 쓰는 것입니다. "2024년 원장 시술 사진"처럼 무엇인지 알 수 있어야 합니다 |
| `links` | **내부 링크 상태** | 홈페이지 내 링크들이 정상적으로 작동하는지 확인합니다 | 깨진 링크가 많으면 검색엔진이 사이트 품질이 낮다고 판단합니다 | 웹 개발자에게 깨진 링크를 수정하고, 주요 페이지 간 내부 링크를 추가해달라고 요청하세요 | 병원 내 안내 표지판이 잘못된 곳을 가리키면 환자가 길을 잃습니다 |
| `eeat_signals` | **전문성/신뢰도 표시** | 의사 프로필, 자격 정보, 환자 후기 등 전문성을 보여주는 콘텐츠가 있는지 확인합니다 | 구글은 의료 분야를 YMYL(건강/재정)로 분류하여 전문성이 없으면 검색 순위를 크게 낮춥니다 | 의사 경력/학력/전문의 자격, 환자 후기, 시술 전후 사진을 홈페이지에 게시하세요 | 병원 로비에 원장 약력과 자격증을 걸어두는 것입니다. 환자가 신뢰할 수 있는 근거를 제공합니다 |
| `structured_data` | **검색 강화 데이터** | 구글에 병원 정보(진료시간, 위치, 전화번호 등)를 구조화된 형태로 제공합니다 | 구글 검색 결과에 진료시간, 별점, 위치 등 풍부한 정보가 표시되지 않습니다 | 웹 개발자에게 "Schema.org의 MedicalClinic 타입 JSON-LD를 추가해달라"고 요청하세요 | 구글 지도에 병원 정보를 등록하는 것의 확장판입니다. 검색 결과에 더 많은 정보가 바로 표시됩니다 |

### 1.4 해외 환자 유치

| 기술 이름 | 고객 친화적 이름 | 한 줄 설명 | 실패 시 영향 | 개선 방법 | 비유 |
|-----------|-----------------|-----------|-------------|-----------|------|
| `multilingual_pages` | **외국어 페이지** | 영어, 일본어, 중국어 등 외국어로 된 페이지가 있는지 확인합니다 | 외국인 환자가 병원 정보를 이해할 수 없어 다른 병원을 선택합니다 | 주요 타겟 국가 언어(영어, 일본어, 중국어)로 핵심 페이지(시술 소개, 가격, 오시는 길)를 번역하세요 | 공항 안내판이 한국어만 있으면 외국인이 길을 찾을 수 없는 것과 같습니다 |

| `overseas_channels` | **해외 메신저 연결** | LINE, WeChat, WhatsApp 등 해외 메신저 채널이 있는지 확인합니다 | 외국인 환자가 익숙한 메신저로 문의할 수 없어 상담 전환율이 떨어집니다 | 타겟 국가에 맞는 메신저(일본: LINE, 중국: WeChat, 동남아: WhatsApp) 공식 계정을 만들고 홈페이지에 링크하세요 | 외국 관광객에게 전화번호만 주면 전화하기 부담스럽지만, 카카오톡처럼 익숙한 앱으로 문의할 수 있으면 부담이 줄어듭니다 |
| `international_search` | **해외 검색엔진 노출** | 일본/대만/동남아 구글, 네이버, 바이두에서 병원이 검색되는지 확인합니다 | 주요 타겟 국가에서 병원이 검색되지 않으면 해외 환자 유입 경로가 차단됩니다 | 타겟 국가 언어로 페이지를 만들고, Google Search Console에서 해당 국가 타겟팅을 설정하세요 | 해외 여행 가이드북에 병원이 실려 있는지 확인하는 것입니다 |

| `content_clarity` | **콘텐츠 명확성** | 페이지 내용이 AI와 검색엔진이 이해하기 좋은 구조인지 확인합니다 | 내용이 불명확하면 AI 검색에서 인용되지 않고, 검색엔진도 관련 키워드에 매칭하지 못합니다 | Q&A 형식의 콘텐츠, 목록형 정리, 충분한 분량(300단어 이상)의 상세 설명을 추가하세요 | 교과서처럼 잘 정리된 내용은 선생님(AI)이 인용하기 쉽지만, 메모장 낙서는 인용할 수 없습니다 |
| `faq_content` | **자주 묻는 질문 (FAQ)** | FAQ 콘텐츠와 구조화 데이터가 있는지 확인합니다 | 구글 검색에서 FAQ 리치결과 표시 기회를 놓치고, AI 검색 인용 확률도 낮아집니다 | 환자들이 자주 묻는 질문 10-20개를 정리하여 FAQ 페이지를 만들고, FAQPage 스키마를 추가하세요 | 병원 안내 데스크에서 자주 받는 질문을 미리 정리해 대기실에 비치하는 것입니다 |

## 2. 카테고리 그룹핑 재설계

### 현재 (AS-IS)
- 24개 항목이 `technical_seo`, `performance` 등 개발자 용어로 flat하게 분류
- 프론트엔드에서 CATEGORY_MAP으로 하드코딩된 2개 카테고리만 차트에 표시
- 나머지 항목(structured_data, multilingual 등)은 개별 key 이름 그대로 노출

### 재설계 (TO-BE)

```
카테고리 구조 (고객 관점 5개 그룹)
├── 검색 노출 기본기 ─ "구글에서 우리 병원이 보이게 하는 기본 설정"
│   ├── robots_txt (5%)
│   ├── sitemap (5%)
│   ├── meta_tags (10%)
│   ├── headings (5%)
│   ├── canonical (3%)
│   └── url_structure (4%)
│   총 가중치: 32%
│
├── 사이트 속도/성능 ─ "환자가 기다리지 않고 바로 볼 수 있는지"
│   ├── lcp (5%)
│   ├── inp (3%)
```

## 3. 실패 유형 분류 체계

### 3.1 유형 정의

| fail_type | 의미 | 누구의 문제인가 | 예시 |
|-----------|------|----------------|------|
| `site_issue` | 홈페이지 자체 문제 | 고객 (수정 필요) | "robots.txt가 존재하지 않습니다", "HTTPS를 사용하지 않습니다" |
| `system_limit` | MediScope 시스템 한계 | 시스템 (입력 데이터 부족) | "병원명 정보가 없어 AI 검색 체크를 수행할 수 없습니다" |
| `api_error` | 외부 API 오류 | 외부 서비스 | "PageSpeed API 접속 불가", "Naver API 오류" |
| `not_applicable` | 해당 없음 / 참고사항 | 해당 없음 | 해외 진출 계획 없는 병원의 다국어/해외 검색 항목 |

### 3.2 유형별 판정 로직 (Worker 측)


| `api_error` | `⬜` | `gray-100` | "외부 서비스 연결 문제로 일시적으로 측정하지 못했습니다. 나중에 다시 확인해 드립니다." |

### 3.4 점수 계산에서의 처리

- `site_issue`: 정상적으로 가중치 포함하여 점수 계산
- `system_limit`: 가중치에서 **제외** (측정 가능한 항목으로만 정규화)
- `api_error`: 가중치에서 **제외** (측정 가능한 항목으로만 정규화)
- `not_applicable`: 가중치에서 **제외**

이렇게 하면 "병원명을 안 넣어서 AI 검색을 못 측정한 건데 0점이 됨" 문제를 해결합니다.

### 5.2 핵심 문제 Top 3 선정 로직

가중치(weight) x (1 - score) 가 가장 큰 3개 항목을 선정합니다.
단, `fail_type`이 `system_limit` 또는 `api_error`인 항목은 제외합니다.

```python
def get_top_issues(category_scores: dict, weights: dict, n: int = 3) -> list:
    """가중 영향도 기준 상위 N개 이슈 추출."""
    issues = []
    for name, data in category_scores.items():
        if data.get("fail_type") in ("system_limit", "api_error", "not_applicable"):
            continue
        if data["score"] is None or data["score"] >= 80:
            continue
        impact = weights[name] * (100 - data["score"])
        issues.append({"name": name, "impact": impact, **data})
    issues.sort(key=lambda x: x["impact"], reverse=True)
    return issues[:n]
```

### 5.3 개선 로드맵 자동 생성 로직

실패 항목을 긴급도/난이도로 분류하여 3단계 로드맵을 자동 생성합니다:

| Phase | 기간 | 포함 항목 (조건) | 설명 |
|-------|------|-----------------|------|
| Phase 1: 즉시 | 당일~3일 | `https`, `mobile`(viewport), `robots_txt` | 설정 변경만으로 해결 가능 |
| Phase 2: 단기 | 1~2주 | `meta_tags`, `sitemap`, `canonical`, `headings`, `images_alt`, `errors_404`, `links` | 웹 개발자 작업 필요 |
| Phase 3: 중기 | 1~3개월 | `structured_data`, `eeat_signals`, `faq_content`, `content_clarity`, `multilingual_pages`, `hreflang`, `overseas_channels`, `performance_score`, `lcp`, `inp`, `cls` | 콘텐츠 작성/번역/최적화 필요 |

점수가 80 이상인 항목은 로드맵에서 제외합니다.
