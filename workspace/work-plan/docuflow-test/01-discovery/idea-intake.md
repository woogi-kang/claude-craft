# Idea Intake: DocuFlow

> **프로젝트**: DocuFlow - 개발자용 API 문서화 자동화 도구
> **작성일**: 2026-01-16
> **Phase**: Discovery > Idea Intake

---

## 1. 아이디어 한 줄 정의

**"코드 주석만으로 10분 안에 프로덕션 레벨 API 문서를 자동 생성하는 개발자 도구"**

---

## 2. 문제 정의 (Problem Statement)

### 2.1 핵심 문제

| 구분 | 내용 |
|------|------|
| **Who** | 스타트업/중소기업 백엔드 개발자 |
| **What** | API 문서 작성에 과도한 시간 소비 |
| **When** | 새 API 개발 후, API 변경 시, 온보딩 시 |
| **Where** | 개발 환경, CI/CD 파이프라인 |
| **Why** | 기존 도구(Swagger)의 복잡한 설정과 학습 곡선 |

### 2.2 문제의 심각도 분석

```

│                                                              │

│   빈도 │       소비             문제                         │
│        │                                                     │
│   중간 │  ★ Swagger 설정   ★ 팀원 온보딩                    │
│        │     복잡성            어려움                        │
│        │                                                     │
│   낮음 │  ★ 외부 연동      ★ 버전 관리                      │
│        │     문서화            혼란                          │
│        ├─────────────────────────────────────────────────────│
│        │     낮음           중간            높음             │
```

### 2.3 현재 대안과 한계

| 현재 대안 | 사용률 | 주요 한계 |
|-----------|--------|-----------|
| **Swagger/OpenAPI** | 70% | 복잡한 YAML 작성, 학습 곡선 높음 |
| **Postman** | 50% | 문서화보다는 테스트 중심, 동기화 어려움 |
| **Notion/Confluence** | 40% | 수동 작성, 코드와 불일치 |
| **README.md** | 80% | 구조화 부족, 유지보수 어려움 |
| **직접 작성 안함** | 20% | 협업 어려움, 기술 부채 |

## 3. 솔루션 가설 (Solution Hypothesis)

### 3.1 핵심 솔루션

```
┌─────────────────────────────────────────────────────────────┐
│                     DocuFlow 솔루션                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   [코드 주석]  ───▶  [파싱 엔진]  ───▶  [문서 생성]         │
│                                                              │
│   /**                                                        │
│    * @api GET /users                                         │
│    * @desc 사용자 목록 조회                                   │
│    * @param limit {number} 조회 개수                          │
│    * @returns {User[]} 사용자 배열                            │
│    */                                                        │
│                                                              │
│              ↓ 자동 변환 ↓                                   │
│                                                              │
│   ┌───────────────────────────────────────────┐             │
│   │  예쁜 웹 문서 + 인터랙티브 테스트 기능     │             │
│   │  - 엔드포인트 목록                         │             │
│   │  - 요청/응답 예시                          │             │
│   │  - Try it out 기능                         │             │
│   └───────────────────────────────────────────┘             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 가설 검증 항목

| 가설 | 검증 방법 | 성공 기준 |
|------|-----------|-----------|
| 개발자는 YAML보다 코드 주석을 선호한다 | 사용자 인터뷰 (N=20) | 80% 이상 선호 |
| 문서화 시간을 70% 단축할 수 있다 | MVP 사용 테스트 | 평균 30분→10분 |
| 기존 코드베이스에 쉽게 적용 가능하다 | 베타 테스트 | 1시간 내 설정 완료 |
| 문서-코드 동기화가 자동으로 된다 | CI/CD 통합 테스트 | 99% 동기화율 |

### 3.3 솔루션 범위 (In/Out Scope)

**In Scope (MVP)**
- JSDoc/TSDoc 스타일 주석 파싱
- REST API 문서 자동 생성
- 웹 기반 문서 뷰어
- GitHub/GitLab CI 연동
- 다크 모드 지원

**Out of Scope (향후)**
- GraphQL 지원
- gRPC 지원
- 다국어 문서화
- 팀 협업 기능 (코멘트 등)
- 버전 비교 기능

## 4. 초기 가정 및 리스크

### 4.1 핵심 가정

| # | 가정 | 신뢰도 | 검증 필요 |
|---|------|--------|-----------|
| A1 | 개발자들은 문서화에 불만이 있다 | 높음 | 설문조사 |
| A2 | 코드 주석 기반이 YAML보다 쉽다 | 중간 | A/B 테스트 |
| A3 | 스타트업은 유료 도구에 비용을 지불한다 | 중간 | 가격 테스트 |
| A4 | CI/CD 연동이 핵심 채택 요소다 | 높음 | 인터뷰 |

### 4.2 주요 리스크

| 리스크 | 영향도 | 발생 가능성 | 대응 전략 |
|--------|--------|-------------|-----------|
| Swagger 생태계의 강력한 점유율 | 높음 | 높음 | 차별화된 UX로 니치 공략 |
| 대형 코드베이스 파싱 성능 | 중간 | 중간 | 증분 파싱 알고리즘 적용 |
| 주석 표준화 부재 | 낮음 | 높음 | 유연한 파서 + 가이드 제공 |
| 경쟁사 기능 카피 | 중간 | 중간 | 빠른 실행 + 커뮤니티 구축 |
