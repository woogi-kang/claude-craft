# DocuFlow - MVP Definition

> 작성일: 2026-01-16
> 버전: 1.0
> 상태: Draft

---

## Executive Summary

DocuFlow MVP는 **"코드 주석 → API 문서 → 원클릭 배포"** 핵심 플로우를 검증합니다. 4주 개발로 핵심 가설을 검증하고, 100명의 베타 사용자 피드백을 수집합니다.

---

## 1. MVP 철학

### 1.1 MVP 정의

```
┌─────────────────────────────────────────────────────────────────┐
│                     MVP = 최소 "사랑받는" 제품                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   "MVP는 최소 기능이 아니라 최소 가치 증명"                        │
│                                                                  │
│   핵심 질문:                                                     │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                          │   │
│   │   사용자가 기존 대안을 버리고                              │   │
│   │   DocuFlow를 선택할 이유가 있는가?                        │   │
│   │                                                          │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│   MVP 범위 결정 기준:                                            │
│   ✓ 핵심 가치 제안 검증 가능                                     │
│   ✓ 4주 이내 개발 가능                                          │
│   ✓ 100명 베타 테스터 확보 가능                                  │
│   ✓ 유료 전환 의향 측정 가능                                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 성공 기준

| 지표 | MVP 목표 | 측정 방법 |
|-----|---------|---------|
| **Activation** | 60% (첫 문서 생성) | 온보딩 완료율 |
| **Time to Value** | 10분 이내 | 가입→첫 문서 시간 |
| **NPS** | 40+ | 주간 설문 |
| **유료 전환 의향** | 30%+ | 설문 조사 |
| **재방문율** | 40%+ | 7일 리텐션 |

│   │                                                          │   │

│                                                                  │
│   가설 2: 시간 가설 (Time Hypothesis)                            │

│   │                                                          │   │
│   │   "개발자는 10분 이내에 첫 API 문서를 생성할 수 있다"          │   │

│   │   검증 지표: 평균 Time to First Doc < 10분                │   │
│   │   검증 방법: 사용자 행동 분석                              │   │
│   │                                                          │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│   가설 3: 지불 가설 (Willingness to Pay)                         │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                          │   │
│   │   "개발자의 30%는 $29/월 이상을 지불할 의향이 있다"           │   │
│   │                                                          │   │
│   │   검증 지표: 유료 전환 의향 설문 > 30%                     │   │
│   │   검증 방법: 가격 민감도 조사                              │   │
│   │                                                          │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

### 2.2 가설 우선순위

| 순위 | 가설 | 위험도 | 검증 우선순위 |
|-----|------|-------|-------------|
| 1 | 가치 가설 | High | 즉시 검증 |
| 2 | 시간 가설 | Medium | 즉시 검증 |
| 3 | 지불 가설 | Medium | 베타 후 검증 |

## 3. 기능 범위 (MoSCoW)

### 3.1 Must Have (MVP 필수)

```
┌─────────────────────────────────────────────────────────────────┐
│                     Must Have - MVP 필수                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   1. 코드 연동                                                   │
│      ├─ GitHub 저장소 연결                                       │
│      ├─ JSDoc/docstring 파싱                                    │
│      └─ 지원 언어: JavaScript, TypeScript, Python               │
│                                                                  │
│   2. 문서 생성                                                   │
│      ├─ AI 기반 자연어 변환                                      │
│      ├─ 기본 테마 (1개)                                         │
│      └─ 실시간 미리보기                                          │
│                                                                  │
│   3. 배포                                                        │
│      ├─ 원클릭 호스팅                                            │
│      └─ docuflow.dev 서브도메인                                  │
│                                                                  │
│   4. 기본 기능                                                   │
│      ├─ 사용자 인증 (GitHub OAuth)                               │
│      ├─ 프로젝트 관리 (CRUD)                                     │
│      └─ 기본 분석 (조회수)                                       │
│                                                                  │
│   개발 예상: 4주                                                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 Should Have (v1.1)

| 기능 | 설명 | 예상 개발 |
|-----|------|---------|
| 추가 언어 지원 | Go, Java, Ruby | 1주 |
| 커스텀 도메인 | CNAME 설정 | 3일 |
| 프리미엄 테마 | 5개 추가 | 1주 |
| API Playground | 인터랙티브 테스트 | 2주 |
| 버전 관리 | 히스토리 저장 | 1주 |

### 3.3 Could Have (v1.2+)

| 기능 | 설명 | 우선순위 |
|-----|------|---------|
| Git Hook 자동 동기화 | 푸시 시 자동 업데이트 | High |
| 팀 협업 | 멀티 유저 | High |
| 다국어 문서 | 자동 번역 | Medium |
| 오프라인 에디터 | 로컬 편집 | Low |

### 3.4 Won't Have (Out of Scope)

| 기능 | 이유 |
|-----|------|
| 온프레미스 설치 | 복잡도 높음 |
| SSO/SAML | Enterprise 전용 |
| API 모니터링 | 범위 외 |
| 코드 생성 | 역방향 기능 |

## 4. 사용자 스토리

### 4.1 핵심 사용자 스토리

```
Epic: 첫 API 문서 생성

US-001: GitHub 연결
"개발자로서, GitHub 계정으로 로그인하여
내 저장소를 DocuFlow와 연결하고 싶다.
그래야 코드를 직접 복사하지 않아도 된다."

Acceptance Criteria:
- [ ] GitHub OAuth 로그인 가능
- [ ] 저장소 목록 조회 가능
- [ ] 저장소 선택 및 연결 가능
- [ ] 연결 해제 가능

---

US-002: 코드 분석
"개발자로서, 연결된 저장소의 API 코드를
자동으로 분석받고 싶다.
그래야 수동으로 엔드포인트를 입력하지 않아도 된다."

Acceptance Criteria:
- [ ] JS/TS/Python 파일 자동 탐지
- [ ] JSDoc/docstring 파싱
- [ ] 엔드포인트 목록 추출
- [ ] 파라미터/응답 타입 추출

---

US-003: AI 문서 생성
"개발자로서, 분석된 코드를
AI가 자연스러운 문서로 변환해주길 원한다.
그래야 기술적인 주석이 읽기 좋은 설명이 된다."

Acceptance Criteria:
- [ ] 기술 용어 → 자연어 변환
- [ ] 사용 예시 자동 생성
- [ ] 에러 응답 설명 생성
- [ ] 실시간 미리보기

---

US-004: 원클릭 배포
"개발자로서, 생성된 문서를
버튼 하나로 웹에 공개하고 싶다.
그래야 호스팅 설정에 시간을 쓰지 않아도 된다."

Acceptance Criteria:
- [ ] 배포 버튼 클릭으로 호스팅
- [ ] xxx.docuflow.dev 도메인 자동 할당
- [ ] HTTPS 자동 적용
- [ ] 배포 상태 표시
```

## 8. 리스크 및 대응

### 8.1 MVP 리스크

| 리스크 | 영향 | 확률 | 대응 |
|-------|-----|------|------|
| AI 품질 불만족 | High | Medium | 프롬프트 튜닝, 수동 편집 옵션 |
| 파싱 정확도 낮음 | High | Medium | 지원 언어 축소, 점진적 확대 |
| 배포 안정성 | Medium | Low | CloudFlare 엔터프라이즈 |
| 베타 참여 저조 | Medium | Low | 인센티브 (Pro 3개월 무료) |

### 8.2 Go/No-Go 기준

```
Go 조건 (모두 충족):
✓ NPS > 30
✓ Activation > 50%
✓ Time to Value < 15분
✓ 유료 전환 의향 > 20%

No-Go 조건 (하나라도 해당):
✗ NPS < 0
✗ Activation < 30%
✗ 핵심 버그 미해결
✗ 베타 사용자 10% 미만 유지
```
