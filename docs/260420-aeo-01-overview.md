## AEO란 무엇인가

**Agentic Engine Optimization (AEO)** 은 기술 콘텐츠를 구조화, 형식화, 제공하여 AI 코딩 에이전트가 실제로 발견하고 사용할 수 있도록 만드는 관행이다.

SEO가 "검색 엔진이 콘텐츠를 잘 찾고 노출하게 만드는 것"이었다면, AEO는 "AI 에이전트가 콘텐츠를 잘 찾고 활용하게 만드는 것"이다.

---

## SEO vs AEO 비교

| 구분 | SEO (검색 엔진) | AEO (AI 에이전트) |
|------|----------------|-------------------|
| 대상 | Google, Bing 크롤러 | Claude Code, Cursor, Cline, Aider |
| 목표 | 검색 결과 상위 노출 | 에이전트 컨텍스트에 정확히 로드 |
| 핵심 제약 | 페이지 랭킹 알고리즘 | 토큰 윈도우 (100K~200K) |
| 발견 파일 | sitemap.xml | llms.txt |
| 권한 파일 | robots.txt | robots.txt + agent-permissions.json |
| 콘텐츠 형식 | 구조화 HTML + Schema.org | 깨끗한 Markdown |
| 성과 측정 | CTR, 순위, 트래픽 | 에이전트 추천률, 통합 성공률 |
| 실패 모드 | 순위 하락 (가시적) | 침묵하는 실패 (비가시적) |

## 왜 지금 AEO가 필요한가

### 1. AI 에이전트 트래픽의 폭발적 증가

- 2025년 3월: Cloudflare 기준 AI 크롤러가 **하루 500억 건 이상** 요청 (전체 웹 트래픽의 ~1%)
- 2026년: API 수요 증가의 **30% 이상**이 AI 도구에서 발생 예상
- GitHub Copilot, Cursor, Claude Code 등 코딩 에이전트가 문서를 직접 소비하는 시대

### 2. 침묵하는 실패 (Silent Failure)

AEO의 가장 위험한 점은 **실패가 보이지 않는다**는 것이다:

- 에이전트가 문서를 못 찾으면 → 할루시네이션으로 대체
- 토큰이 초과되면 → 조용히 잘라내거나 완전 스킵
- robots.txt가 차단하면 → 아무 알림 없이 접근 불가
- 개발자는 "왜 AI가 우리 API를 잘못 사용하는지" 원인을 모름

### 3. 에이전트 최적화 = 인간 문서도 개선

AEO를 위한 작업이 인간 독자에게도 긍정적 효과를 준다:

- 간결하고 구조화된 문서 → 인간도 빠르게 스캔
- 결과-먼저 구조 → 인간의 F-패턴 읽기에도 유리
- 토큰 수 표시 → 문서 크기 인식 향상
- Markdown 가용성 → IDE 내 빠른 참조

## AEO의 5가지 핵심 축

```
┌─────────────────────────────────────────────┐
│               AEO 5대 요소                    │
├──────────────┬──────────────────────────────┤
│ Discoverability │ 에이전트가 문서를 발견할 수 있는가? │
│ Parsability     │ 기계가 읽을 수 있는 형식인가?      │
│ Token Efficiency│ 컨텍스트 윈도우에 들어가는가?      │
│ Capability Signal│ 무엇을 할 수 있는지 명시하는가?   │
│ Access Control  │ 접근 권한이 올바른가?              │
└──────────────┴──────────────────────────────┘
```

### Discoverability (검색 가능성)
- JavaScript 렌더링 없이 에이전트가 문서를 발견 가능해야 함
- `llms.txt`로 구조화된 사이트맵 제공
- `AGENTS.md`로 프로젝트 진입점 제공

### Parsability (파싱 가능성)
- 시각적 레이아웃 해석이 불필요한 머신 리더블 콘텐츠
- HTML 대신 Markdown 제공 (90% 토큰 절감)
- 네비게이션 노이즈 (사이드바, 이동경로, 바닥글) 제거

### Token Efficiency (토큰 효율성)
- 전형적 에이전트 컨텍스트 윈도우(100K~200K 토큰) 내 수용
- 페이지당 토큰 수 측정 및 공개
- 빠른 시작: <15K, API 참조: <25K, 개념 가이드: <20K

### Capability Signaling (기능 신호)
- API가 "무엇을 하는가" 명확히 표시 (`skill.md`)
- 성취 가능한 것, 필수 입력, 제약사항을 구조화
- "어떻게 호출하는가"보다 "무엇이 가능한가"에 초점

### Access Control (접근 제어)
- `robots.txt` 적절한 구성 (무의식적 차단 방지)
- `agent-permissions.json`으로 자동화 상호작용 규칙 선언
- 속도 제한, 선호 엔드포인트 명시

## 인간 vs 에이전트의 문서 소비 패턴

### 인간의 패턴
```
1. 문서 홈페이지 방문
2. 관련 섹션으로 네비게이션
3. 헤딩 스캔
4. 코드 샘플 읽기/실행
5. 내부 링크 따라가기
→ 세션 시간: 4~8분
→ 분석 추적 가능 (스크롤, 클릭, 체류시간)
```

### 에이전트의 패턴
```
1. 1~2개 HTTP 요청으로 멀티페이지 탐색 압축
2. 전체 페이지 수신 후 즉시 이동
3. 클라이언트측 분석 이벤트 무시
→ 세션 시간: 밀리초 단위
→ 스크롤 깊이, 버튼 클릭, 완료 시간 모두 0
```

### 개발자 경험의 패러다임 전환

| 기존 (인간 중심) | AEO (에이전트 중심) |
|-----------------|-------------------|
| 시각적 계층 중요 | 시각적 계층 무관 |
| 점진적 공개 (Progressive Disclosure) | 오히려 장애물 |
| 인터랙티브 예시 | 가치 감소 |
| 사용자 여정 설계 | 단일 컨텍스트 로드로 축약 |
| 클릭 깊이 최적화 | HTTP 요청 수 최소화 |

## 누가 이점을 얻는가

### 초기 진출 팀 (Early Movers)

1. **에이전트가 권장하는 API** — 에이전트가 경쟁사 대신 자사 API를 추천
2. **높은 통합 성공률** — 에이전트가 정확한 코드를 생성하므로 개발자 온보딩 단축
3. **재방문 감소** — 한 번에 올바른 통합 → 디버깅/질문 감소
4. **지원 비용 절감** — 할루시네이션 기반 코드로 인한 지원 티켓 감소

### 측정 가능한 성과

- AI 추천 트래픽 비율 (Referrer 분석)
- 에이전트 기반 API 호출 성공률
- 문서 관련 지원 티켓 감소율
- 에이전트 통합 완료율
